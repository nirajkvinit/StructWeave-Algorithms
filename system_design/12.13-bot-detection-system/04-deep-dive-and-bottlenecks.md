# 04 — Deep Dives & Bottlenecks: Bot Detection System

## Deep Dive 1: Behavioral Analysis Engine

The behavioral analysis engine is the most signal-rich and computationally demanding component of the detection system. It must process raw event streams from millions of concurrent browser sessions, extract meaningful features in real time, and produce a continuous behavioral score that evolves as the session progresses.

### Event Ingestion Architecture

Behavioral events arrive as compressed protocol buffers via HTTPS POST from the JavaScript collector running in each browser. The ingestion layer is designed as a stateless, horizontally scaled fleet of receivers that immediately publish events to a partitioned event stream, keyed by session ID. This partitioning ensures all events for a single session are processed by the same downstream consumer, enabling stateful feature computation without distributed coordination.

```
Browser JS Collector
    ↓  (POST /v1/beacon, every 30sec or 1000 events)
Beacon Receiver Fleet  (10,000 nodes globally, auto-scaled)
    ↓  (publish to partitioned event stream)
Behavioral Feature Workers  (keyed by session_id)
    ↓  (update running feature aggregates)
Session Score Updater
    ↓  (write to session store)
Risk Decision Engine  (reads updated session state)
```

### Feature Computation: Streaming Aggregates

Rather than storing raw events (which would require 1–2TB/day per million active sessions), the behavioral engine computes rolling aggregates in real time:

**Mouse features (updated per beacon):**
- Mean and variance of velocity between consecutive points
- Distribution of turn angles (entropy of direction changes)
- Proportion of movements along perfectly horizontal or vertical lines
- Acceleration profile: histogram of speed change rates
- Click-to-element targeting precision (random vs. purposeful clicking)
- Overshoot-correction patterns (human mouse often overshoots target then corrects)
- Idle time distribution between active movement bursts

**Keystroke features (updated per beacon):**
- Dwell time distribution: mean, variance, skewness
- Flight time distribution: mean, variance, coefficient of variation
- Bigram timing consistency: for each repeated two-key combination, timing variance
- Typing speed trend: is speed increasing (warming up) or constant (bot)?
- Error pattern: backspace frequency and corrective behavior

**Scroll features (updated per beacon):**
- Scroll velocity profile (humans scroll in irregular bursts)
- Scroll entropy: randomness of scroll position sequence
- Viewport engagement: proportion of page visible vs. requested

### The Adversarial Behavioral Mimicry Problem

The most sophisticated bots in 2025–2026 use synthesized mouse trajectories generated by neural networks trained on real human interaction recordings. These "human-like" trajectories can fool naive feature detectors because they reproduce the broad statistical properties of human movement (curvature, velocity distribution, micro-jitter).

The defense requires detecting second-order artifacts:

1. **Trajectory smoothness anomaly:** Synthesized paths are generated by neural networks that optimize for plausibility. They tend to be slightly *too smooth*—the variance of their curvature is lower than real human paths, which have unpredictable micro-corrections.

2. **Temporal correlation across sessions:** If a bot farm generates trajectories from the same generative model, sessions from different IPs will show statistically similar trajectory statistics. Clustering session behavioral features across IP space reveals bot farms even when individual sessions look plausible.

3. **Interaction-content mismatch:** A human reading a page and a bot extracting content show different interaction patterns relative to the page's content structure. A user who never moves their mouse near any clickable element, or who interacts with elements in DOM order rather than visual layout order, is exhibiting bot-like behavior.

4. **Reaction time plausibility:** Real human click reactions to page events have a minimum latency of ~150ms (visual processing + motor execution). A bot responding to a dynamically injected element in < 50ms reveals itself regardless of how natural its mouse path looks.

## Deep Dive 2: Device Fingerprinting System

### The Fingerprint as a Stability Signal

Device fingerprinting serves two distinct purposes in bot detection:

1. **Identity continuity:** A returning device can be recognized even without cookies, allowing session reputation to transfer across cookie-cleared browsing or new browser tabs.

2. **Anomaly detection:** The fingerprint itself contains internal consistency checks—a browser claiming to be Chrome on Windows but reporting macOS-only WebGL renderer strings, or having WebGPU support that was only introduced in a version later than the reported user-agent, reveals inconsistency that is a strong bot signal.

### Multi-Layer Fingerprint Architecture

```
Layer 1: Passive Signals (from HTTP headers)
├── User-Agent string parsing
├── Accept-Language header
├── Accept-Encoding header
├── HTTP/2 SETTINGS frame ordering
└── TLS fingerprint (JA4 hash)
   → Low cost, always available, easily spoofed

Layer 2: Active JS Probes (executed by injected JavaScript)
├── navigator.* API enumeration
├── Screen dimensions and color depth
├── Timezone resolution
├── Plugin and MIME type lists
└── Feature detection (payment API, battery API, etc.)
   → Medium cost, runs in < 50ms, moderately spoofable

Layer 3: Hardware-Bound Signals (GPU/CPU/Audio rendering)
├── Canvas 2D rendering hash
├── WebGL renderer and parameter hash
├── AudioContext oscillator hash
├── WebGPU adapter info (2025+ browsers)
└── Performance timing of crypto operations
   → High cost (100–500ms), hard to spoof consistently

Layer 4: Behavioral Fingerprint (accumulated over session)
├── Mouse trajectory statistical signature
├── Typing rhythm signature
└── Scroll behavior signature
   → Very high cost, requires session time, hardest to fake
```

### Fingerprint Stability and Collision Analysis

A fingerprint hash must be stable enough to re-identify the same device across sessions, but unique enough to distinguish different devices. Analysis of production fingerprint databases shows:

- **Canvas + WebGL hash** alone achieves ~95% device uniqueness (1 in 20 devices share a canvas hash if from same hardware batch)
- Adding **AudioContext hash** increases uniqueness to ~99.5%
- Adding **navigator.* enumeration** pushes uniqueness to ~99.9%
- Full Layer 2+3 fingerprint: uniqueness > 99.99%

However, stability degrades over time as browsers update (canvas rendering changes with browser updates every 6 weeks). The fingerprint system must account for temporal drift:

```
FUNCTION fingerprint_match(new_fp, stored_fp, days_since_stored):
    base_similarity = cosine_similarity(new_fp.feature_vector, stored_fp.feature_vector)

    // Allow for browser update drift: reduce required similarity threshold over time
    drift_allowance = min(0.1, days_since_stored * 0.005)  // max 10% drift
    similarity_threshold = 0.90 - drift_allowance

    // But hardware-bound signals should NOT change with browser updates
    hardware_similarity = compare_hardware_signals(new_fp, stored_fp)

    IF base_similarity >= similarity_threshold AND hardware_similarity >= 0.95:
        RETURN MATCH
    ELSE IF base_similarity >= 0.7 AND hardware_similarity >= 0.95:
        RETURN PROBABLE_MATCH  // same hardware, browser updated
    ELSE:
        RETURN NO_MATCH
```

### Anti-Detect Browser Evasion and Counter-Detection

Anti-detect browsers (used by sophisticated bot operators) override canvas, WebGL, and AudioContext APIs to return pre-configured, plausible fingerprints. Detection of these tools relies on:

1. **API timing analysis:** Overriding canvas rendering makes `toDataURL()` calls either faster (if returning a pre-computed value) or slower (if running through a software renderer). Measuring the time to render a specific scene and comparing against hardware-specific baselines reveals software rendering.

2. **Internal consistency of the fingerprint stack:** Anti-detect browsers must fake dozens of correlated attributes. Inconsistencies appear: a claimed RTX 3090 GPU that renders a WebGL scene at performance equivalent to an integrated GPU, or font rendering that doesn't match the reported operating system's font smoothing settings.

3. **WebGPU as an escalation signal:** WebGPU (introduced in stable Chrome 113, 2023) exposes much more detailed hardware information than WebGL. An anti-detect browser that claims to support WebGPU but provides implausible adapter info, or one that doesn't support WebGPU despite claiming to be Chrome 120+, reveals itself.

## Deep Dive 3: ML Scoring Architecture

### Two-Tier Model Design

The core tension in ML-based bot detection is between latency and accuracy. A single model cannot serve both constraints:

**Edge model (Tier 1):**
- Architecture: Gradient-boosted tree (GBT), 100 features, 100 trees with depth 6
- Size: ~5MB, fits in CPU L2/L3 cache across all edge PoPs
- Latency: 0.5–2ms inference
- Training: 100B labeled samples from past 30 days
- Accuracy: ~88% AUC on held-out set
- Deployed to: All CDN edge nodes (5,000+ globally)

**Deep model (Tier 2):**
- Architecture: Gradient-boosted ensemble + shallow neural net, 500 features
- Size: ~200MB, GPU-resident
- Latency: 10–30ms inference
- Training: Full 500-feature set including sequence features, cross-session statistics
- Accuracy: ~96% AUC on held-out set
- Deployed to: Regional cloud inference clusters (10–20 regions)
- Invoked for: Requests where edge score falls in [0.25, 0.75] range (borderline)

### Feature Engineering Pipeline

Features are computed in stages with increasing computational cost:

```
Stage 1 (always computed, < 0.1ms):
  - IP numeric features (is_datacenter, is_tor, reputation_score, country_code_one_hot)
  - User-agent parse features (browser_family, os_family, major_version)
  - Request metadata (hour_of_day, day_of_week, endpoint_category)
  - Session age, request count

Stage 2 (if session state available, < 0.5ms):
  - Session risk score history (last 10 scores)
  - Session IP diversity index
  - Challenge history (count of challenges issued, solved, failed)
  - Time-since-last-human-verification

Stage 3 (if fingerprint available, < 1ms):
  - Fingerprint consistency score
  - Fingerprint age and session count
  - Hardware anomaly flags (webdriver, chrome_runtime, etc.)
  - IP diversity index for this fingerprint

Stage 4 (if behavioral data available, < 2ms):
  - Mouse trajectory statistics (7 features)
  - Keystroke dynamics statistics (6 features)
  - Scroll entropy and engagement metrics (4 features)
  - Interaction sequence features (timing between page events)

Stage 5 (Tier 2 only, < 10ms):
  - Cross-session behavioral clustering features
  - Temporal pattern features (request timing distribution)
  - Network graph features (IP-fingerprint-session co-occurrence)
  - Threat intelligence correlation scores
```

### Model Training and Label Generation

The hardest problem in bot detection ML is label quality. Ground truth labels come from multiple noisy sources:

1. **Challenge outcomes:** Sessions that fail interactive CAPTCHAs are labeled bot. Sessions that solve them are labeled human. Confidence: high, but CAPTCHA solvers (human-farms) contaminate the "human" label.

2. **Honeypot signals:** Invisible links and form fields that only bots would interact with provide high-confidence bot labels. Confidence: very high for positive labels, no negative label signal.

3. **Account behavior correlation:** Sessions that subsequently exhibit known fraud (credential stuffing attempts, account takeover) are retroactively labeled bot. Confidence: high but delayed (24-48 hour lag).

4. **Operator reports:** Customer-confirmed false positives provide human labels. Confidence: moderate (operators can be wrong).

5. **Known crawler allowlist:** Sessions from verified search engine IP ranges with matching TLS fingerprints provide high-confidence benign bot labels.

```
FUNCTION build_training_labels(sessions_from_past_30_days):
    labels = {}

    FOR session IN sessions:
        confidence = 0.0
        label = NULL

        IF session.challenge_state == CAPTCHA_FAILED:
            label = BOT
            confidence = 0.85  // CAPTCHA solvers contaminate ~15%

        IF session.triggered_honeypot:
            label = BOT
            confidence = 0.99

        IF session.subsequent_fraud_detected:
            label = BOT
            confidence = 0.90

        IF session.operator_confirmed_false_positive:
            label = HUMAN
            confidence = 0.95

        IF session.is_verified_search_crawler:
            label = BENIGN_BOT
            confidence = 0.99

        IF session.challenge_state == CAPTCHA_PASSED:
            label = HUMAN
            confidence = 0.80  // CAPTCHA farm contamination ~20%

        IF label != NULL AND confidence >= 0.80:
            labels[session.id] = WeightedLabel(label, confidence)

    RETURN labels  // ~30-40% of sessions get labels; rest unlabeled
```

## Deep Dive 4: Challenge-Response System

### Progressive Challenge Ladder

The challenge system operates as a "progressive trust escalation" ladder:

```
Risk Score    Challenge Type          User Experience         Bot Solve Rate
─────────────────────────────────────────────────────────────────────────────
0.0 – 0.2    None (allow)            Invisible               N/A
0.2 – 0.35   JS Probe (invisible)    Page loads normally     ~80% (fails env checks)
0.35 – 0.5   Proof of Work           50-500ms CPU delay      ~50% (too slow to be useful)
0.5 – 0.7    Interactive CAPTCHA     5-30 sec user task      <5% for sophisticated tasks
0.7 – 1.0    Hard Block              HTTP 403                0% (no path forward)
```

### Proof-of-Work Design

The PoW challenge assigns a computational puzzle that is expensive for bots running at scale but imperceptible to individual users:

```
Challenge:
  nonce = random_bytes(16)
  difficulty = dynamic_difficulty(current_bot_threat_level)
    // Standard: 18 bits (2^18 = 262,144 hashes, ~200ms on laptop)
    // High threat: 22 bits (2^22 = 4M hashes, ~3sec on laptop)
    // Under DDoS: 24 bits (2^24 = 16M hashes, ~12sec on laptop)

Client task:
  FIND solution such that:
    SHA256(nonce || solution) has `difficulty` leading zero bits

Cost analysis:
  - For a human user: 200ms is imperceptible
  - For a bot farm running 1,000 concurrent sessions on a server:
    1,000 × 200ms CPU = 200 CPU-seconds/sec = 200 continuous cores
    This makes large-scale botting economically expensive
```

### Interactive CAPTCHA Design Principles

Modern interactive CAPTCHAs in 2025–2026 have moved beyond "select all traffic lights" (now fully solvable by vision models). Production systems use:

1. **Semantic reasoning tasks:** "Select the object that could contain liquid" requires understanding of physical properties, not just visual classification. Current vision models solve these at ~60% accuracy.

2. **3D rotation tasks:** "Rotate the object until the arrow points up" combines spatial reasoning with fine motor interaction. The interaction itself (how the user drags to rotate) is also a behavioral signal.

3. **Short-term memory tasks:** "Remember these 3 objects, then select them from a larger grid shown 3 seconds later." Requires sequential memory, challenging for stateless models.

4. **Adversarial image puzzles:** Images with perturbations that are imperceptible to humans but confuse computer vision models. The system periodically retests with new perturbation types to outpace adversarial ML defenses.

### Challenge Solve Event Analysis

Even when a challenge is solved correctly, the interaction pattern during solving is a secondary behavioral signal:

```
FUNCTION analyze_captcha_solve_events(solve_events, task_type):
    // solve_events: list of (event_type, x, y, timestamp) during CAPTCHA interaction

    // Check 1: Total solve time plausibility
    total_time = solve_events[-1].t - solve_events[0].t
    IF total_time < MIN_HUMAN_SOLVE_TIME[task_type]:
        RETURN BOT  // too fast even for the easiest task

    // Check 2: Interaction density (humans interact more than minimum needed)
    click_count = count_where(e: e.type == CLICK)
    IF click_count == minimum_required_clicks:
        score += 0.3  // bot-like efficiency: clicks exactly what's needed

    // Check 3: Mouse path during selection
    path_to_first_click = events_before_first_click()
    cursor_linearity = compute_path_linearity(path_to_first_click)
    IF cursor_linearity > 0.99:  // perfectly straight path to target
        score += 0.4

    // Check 4: Pause patterns (humans often pause before clicking)
    pauses = [intervals where no mouse movement for > 200ms]
    IF len(pauses) == 0:
        score += 0.3  // bots don't pause to "think"

    RETURN score >= 0.7 ? BOT : HUMAN
```

## Race Conditions and Edge Cases

### Challenge Token Race Condition

**Problem:** Two concurrent requests from the same session can both receive a challenge token, and a bot could attempt to solve the cheaper one (PoW) while ignoring the harder one (CAPTCHA), then replay the PoW token for both requests.

**Solution:** Challenge tokens are bound to both `session_id` and `request_id`. The verification service uses optimistic locking: the token record includes a `solved` boolean that is set atomically via compare-and-swap. Attempting to use an already-solved token returns an error and immediately escalates the session to the next challenge tier.

### Session Store Partition Failure

**Problem:** The session store (distributed cache) is partitioned, and a subset of session IDs become unreadable. Edge nodes cannot retrieve session state for affected sessions.

**Solution:** Edge nodes detect partition via cache read errors and switch to stateless evaluation for affected sessions: they use only per-request signals (IP, fingerprint, headers) without session history. The neutral prior (0.5 score) is used as baseline. This increases false positive rate temporarily but does not cause widespread blocking.

### Fingerprint Collision Between Human and Bot

**Problem:** Two devices (one human, one bot) produce an identical canvas fingerprint hash. The bot's session elevates the fingerprint's `bot_probability`, which then causes the human's requests to be scored higher.

**Solution:** The fingerprint's `bot_probability` is capped at contributing 40% of the final score. The remaining 60% comes from behavioral signals and network signals, which differ between the human and bot even with identical fingerprints. Additionally, `ip_diversity` (number of distinct IPs using this fingerprint) is tracked: if a fingerprint is seen from >100 distinct IPs, it is likely a spoofed/cloned fingerprint, which is itself a strong bot signal.

## Bottleneck Analysis

### Bottleneck 1: Session Store Read Latency at Edge

**Problem:** The session store must be readable from every CDN edge PoP with < 1ms latency. A global distributed cache (even with replication to 50 regions) has P99 inter-region reads of 10–50ms due to light-speed propagation.

**Solution:** Two-tier session storage:
- **Local edge cache:** In-process LRU cache per edge node, holds recently seen session states (TTL 30 seconds). Cache hit rate: ~85% for returning sessions within a 30-second window.
- **Regional session store:** Per-region cache cluster, reads: < 2ms intra-region. Cache hit rate for regional sessions: ~95%.
- **Global session store:** Globally replicated, written on session updates, read only on complete cache miss. < 50ms but used rarely.

### Bottleneck 2: Behavioral Feature Worker State Fan-Out

**Problem:** Behavioral feature workers process event streams keyed by session ID. A session receiving 200 mouse events/second generates a 200 msg/sec stream to the same partition. A single worker processing this partition must handle the full event rate for all sessions assigned to it.

**Solution:** Two-level partitioning:
- Events are first partitioned by session_id to maintain ordering
- Within each partition, sessions are processed in parallel using goroutine-per-session pools
- Feature state is kept in-process (not in a remote store) to avoid network round-trips per event
- State is checkpointed to the session store every 10 seconds rather than per-event

### Bottleneck 3: ML Model Update Propagation

**Problem:** Updating the edge model requires pushing ~5MB to 5,000+ edge nodes. A naive sequential push would take hours; a simultaneous push would overload the model registry.

**Solution:** BitTorrent-inspired hierarchical distribution:
- Model registry serves model to 50 "seed" edge nodes (one per major region)
- Each seed node serves the model to ~100 neighboring edge nodes
- Propagation completes within 3–5 minutes globally
- Edge nodes validate model hash before activating; corrupted downloads retry from different seed

### Bottleneck 4: Beacon Receiver Ingestion Surge

**Problem:** Major bot attacks correlate with attack campaigns: suddenly thousands of bot sessions all start sending high-frequency beacons simultaneously, creating a 10–100x spike in the ingestion layer.

**Solution:** Beacon receivers are deployed with horizontal auto-scaling triggered by queue depth. Additionally, the beacon payload is rate-limited per session: if a session sends more than 50 beacons/minute, the server returns a `next_beacon_ms` directive of 60,000ms (1 minute), throttling the session's reporting rate. Bot sessions (which don't respect this directive) are flagged as an additional signal.
